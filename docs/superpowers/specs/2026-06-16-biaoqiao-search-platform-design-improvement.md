# 美术标签搜索平台 - 方案改进建议

## 结论摘要

原方案的总体架构方向合理：PostgreSQL 作为数据真相源，Elasticsearch 承担搜索和聚合，`tag_definitions` 驱动前端筛选面板、查询构建和 LLM 解析上下文。这套设计适合美术资产标签维度持续扩展的场景。

建议在进入实现前补强 4 类内容：

- 安全边界：明确内网部署、管理接口鉴权、ES/Redis/PG 暴露范围和 LLM 外发数据边界。
- 数据一致性：补充 PG 与 ES 同步失败、重试、重建索引和回滚策略。
- 工程验收：将性能描述从“预估值”改为可压测验证的 P95 指标。
- 一期范围：收敛 MVP，避免首版同时承担过多高级能力。

---

## 必须修改项

### 1. 修正 `pg_trgm` 设计矛盾

原文在“数据库扩展”中说明 PG 不再需要 `pg_trgm`，但 `search_logs` 表又创建了以下索引：

```sql
CREATE INDEX idx_search_logs_query ON search_logs USING GIN (raw_query gin_trgm_ops);
```

该索引依赖 `pg_trgm` 扩展。建议二选一：

- 如果需要搜索日志模糊查询：在初始化 SQL 中补充 `CREATE EXTENSION IF NOT EXISTS pg_trgm;`。
- 如果一期不查询搜索日志：删除该索引，后续做热门搜索或日志分析时再补。

推荐一期删除该索引，避免为非核心能力增加数据库扩展依赖。

### 2. 收紧 Elasticsearch 暴露和安全配置

当前 Docker Compose 示例中关闭了 ES 安全能力，并暴露了 `9200` 端口：

```yaml
environment:
  - xpack.security.enabled=false
ports: ["9200:9200"]
```

即使部署在公司内网，也不建议让 ES 直接暴露给使用者网络。建议改为：

- ES、PG、Redis 仅在 Docker 内部网络访问。
- 宿主机只暴露 Nginx 的 HTTP/HTTPS 端口。
- 管理端接口必须鉴权，至少包括：
  - `/admin/import-excel`
  - `/admin/reindex-es`
  - `/admin/refresh-dictionary`
  - `/tags/synonyms`
- 如果确实需要调试 ES，使用临时 SSH 隧道或仅绑定 `127.0.0.1`。

### 3. 明确 LLM 外发数据边界

原方案允许 LLM 解析请求走外网，但未说明哪些数据可以发送。美术资源路径、内部命名和标签可能属于公司内部信息，应补充数据边界：

- 默认只发送“剩余未匹配文本”和必要的标签候选值。
- 不发送资源路径、文件名、完整标签库、用户身份信息。
- 对 prompt 和响应做日志脱敏。
- LLM 调用失败不能影响主搜索链路，必须降级为关键词搜索。
- 提供关闭 LLM 的配置开关，例如 `LLM_PARSE_ENABLED=false`。

### 4. 补充 PG 与 ES 同步失败处理

原方案写了“PG 写入后同步 ES”，但没有定义同步失败时的恢复方式。建议补充以下机制：

- 批量导入时记录 `batch_id`，每批 ES bulk index 的成功数和失败数可追踪。
- 建议增加同步状态字段或单独 outbox 表：

```sql
CREATE TABLE asset_index_events (
    id              BIGSERIAL PRIMARY KEY,
    asset_id        BIGINT NOT NULL,
    event_type      VARCHAR(20) NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'pending',
    retry_count     INTEGER NOT NULL DEFAULT 0,
    error_message   TEXT,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);
```

- ES 写入失败时进入重试队列，不让 PG 数据回滚。
- 提供后台任务重试 pending/failed 事件。
- 提供全量重建索引接口作为最终修复手段。

### 5. 使用 ES alias 做无损重建

`/admin/reindex-es` 不应直接覆盖线上索引。建议采用 alias 切换：

1. 当前线上 alias 为 `assets_current -> assets_v1`。
2. 重建时创建 `assets_v2`。
3. 从 PG 全量写入 `assets_v2`。
4. 校验文档数、抽样查询和 mapping。
5. 原子切换 alias：`assets_current -> assets_v2`。
6. 保留旧索引一段时间，便于回滚。

这样重建过程中线上查询不受影响，也能降低误操作风险。

## 建议补充项

### 1. 将性能预估改为验收指标

原文中的 `< 20 ms`、`< 10 ms` 更像理想估计。建议改为可压测的验收标准：

| 场景 | 建议验收指标 | 说明 |
|------|--------------|------|
| 纯筛选查询 | P95 < 100 ms | 不含网络静态资源加载 |
| 关键词搜索 | P95 < 200 ms | 包含 IK 分词、highlight |
| 筛选 + 聚合计数 | P95 < 300 ms | facets 较多时允许更高 |
| 词典解析 | P95 < 10 ms | 应用内存匹配 |
| LLM 首次解析 | P95 < 3 s | 超时后降级 |
| 缓存命中解析 | P95 < 20 ms | Redis 命中 |
| Excel 导入 | 10 万条可在 10 分钟内完成 | 视图片提取量调整 |

建议使用 k6 或 Locust 编写压测脚本，固定 10 万条测试数据后验收。

### 2. 收敛一期 MVP 范围

当前方案覆盖面较广，建议明确一期只交付核心闭环：

- Excel 导入模型、动作、特效基础数据。
- PG 入库和 ES 索引同步。
- 模块切换、动态筛选、关键词搜索、分页排序。
- 模型截图和特效 GIF 预览。
- 词典匹配解析自然语言。
- 管理端刷新词典、重建索引、查看导入错误。

建议暂缓到二期：

- LLM 解析。
- 收藏。
- 热门搜索和搜索历史利用。
- 搜索纠错。
- Prometheus + Grafana。
- 多副本部署。

这样一期更容易按时落地，也能尽早拿真实美术团队的搜索反馈。

### 3. 明确 ES mapping 生成策略

当前方案依赖 `dynamic_templates` 自动映射 `tags.*`，实现简单，但存在类型冲突风险。例如 `duration` 第一次写入 `"5s"` 会被识别为 keyword，后续写入数字 `5.0` 就会冲突。

建议改为：

- 导入前根据 `tag_definitions.field_type` 强校验类型。
- 由后端基于 `tag_definitions` 生成 ES mapping。
- `number_range` 统一写 JSON number。
- `boolean` 统一写 true/false。
- 枚举和多选统一写 keyword 或 keyword 数组。
- mapping 变更时通过新索引 + alias 切换生效。

如仍保留动态映射，也应在文档中明确“第一次写入前必须完成数据类型归一化”。

### 4. 补充标签冲突和歧义处理

词典匹配中提到“同优先级多字段命中降级为 keyword”，建议进一步明确：

- 同义词优先级由谁维护。
- 同一个词命中多个模块时是否按当前 `module_type` 过滤。
- 面板筛选和 NLP 解析冲突时，前端如何展示。
- 被覆盖的解析标签是否进入 `ignored_tags`。

推荐规则：

- 当前模块优先。
- 手动筛选优先于 NLP 解析。
- 高置信度词典命中优先于 LLM。
- 低置信度结果只作为 keyword，不自动变成筛选条件。

### 5. 补充前端关键状态

前端组件树已经清楚，但建议补充几个用户一定会遇到的状态：

- 空结果：展示当前筛选条件，并提供“清空关键词”“放宽筛选”“重置全部”。
- 解析失败：提示智能解析不可用，但保留普通关键词搜索结果。
- 解析冲突：展示某些 NLP 标签已被手动筛选覆盖。
- 导入中：管理端展示导入进度、成功数、失败数、未知标签数。
- 预览缺失：统一占位图，避免卡片布局跳动。

### 6. 补充备份和恢复策略

建议一期就补充：

- PG 每日备份，至少保留 7 天。
- 预览文件目录定期备份。
- ES 不作为主备份对象，可从 PG 重建。
- 导入前可生成批次快照或记录 batch，支持按批次回滚。
- 配置文件和标签初始化 SQL 纳入 Git 管理。

### 7. 增加可观测性

即使不接 Prometheus，也建议保留基础日志指标：

- 每次搜索的 `query_time_ms`、`parse_time_ms`、`result_count`。
- LLM 调用耗时、失败原因、降级次数。
- ES bulk index 成功数、失败数。
- Excel 导入耗时、失败行、未知标签。
- 慢查询日志阈值，例如超过 500 ms 记录 ES query DSL。

## 文档结构改进建议

### 1. 增加“非目标”章节

建议明确一期不做什么，减少实现过程中的范围漂移：

- 不做权限体系细分，只做管理接口基础鉴权。
- 不做跨部门资产权限隔离。
- 不做复杂推荐算法。
- 不做资源文件在线编辑。
- 不保证 LLM 解析 100% 准确，LLM 只做增强。

### 2. 增加“里程碑计划”

建议拆成 4 个阶段：

| 阶段 | 目标 | 交付物 |
|------|------|--------|
| P0 | 数据闭环 | PG schema、Excel 导入、导入错误记录 |
| P1 | 搜索闭环 | ES 索引、筛选查询、关键词搜索、分页 |
| P2 | 前端闭环 | 动态筛选面板、结果卡片、预览展示 |
| P3 | 体验增强 | 词典解析、搜索建议、管理端重建索引 |

LLM 解析建议放在 P3 之后作为增强项，避免阻塞主链路。

### 3. 增加“验收清单”

建议在文档末尾加入：

- [ ] 10 万条测试数据导入成功。
- [ ] 导入失败行可在 `import_errors` 查询。
- [ ] 模型、动作、特效 3 个模块可以切换搜索。
- [ ] 筛选项全部来自 `tag_definitions`，前端无标签维度硬编码。
- [ ] 修改标签配置后，刷新词典即可生效。
- [ ] ES 索引可从 PG 全量重建。
- [ ] LLM 关闭时，普通搜索仍可用。
- [ ] 管理接口需要鉴权。
- [ ] PG 和预览文件具备备份策略。
- [ ] 压测达到约定 P95 指标。

---

## 推荐实施顺序

1. 先修正文档中的 `pg_trgm` 矛盾、安全暴露和 LLM 数据边界。
2. 再补 PG 与 ES 同步失败处理、alias 重建索引和备份恢复策略。
3. 然后将性能指标改成可压测验收标准。
4. 最后收敛一期 MVP，并把 LLM、收藏、热门搜索、监控告警移到二期或增强项。

如果按这个顺序改，方案会从“完整架构设想”变成“可以开工、可以验收、出问题能恢复”的工程方案。
